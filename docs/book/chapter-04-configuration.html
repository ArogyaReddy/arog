Level 1: Organization

org-config.json

What It Is: Company-wide standards that apply to ALL projects.

When to Use:

  • CTO wants consistent standards across all teams
  • Security team mandates vulnerability thresholds
  • Compliance requires accessibility standards (WCAG 2.1 AA)
  • Finance enforces cost optimization
{ "standards": { "codeQuality": { "eslint": "airbnb", "prettier": true, "typescript": "strict" }, "testing": { "minCoverage": 80 }, "security": { "vulnerabilityThreshold": "moderate" } } }

AROG Benefits:

  • Enforces standards automatically across all 30 projects
  • No manual updates needed
  • Guaranteed compliance
  • Single source of truth
ğŸ‘¥

Level 2: Team

team-config.json

What It Is: Team-specific rules that override organization config.

When to Use:

  • Frontend team needs React-specific linting
  • Backend team requires API testing
  • Team lead wants higher coverage than org minimum
{ "extends": "org-config.json", "team": { "name": "Frontend Team" }, "overrides": { "testing": { "minCoverage": 90 // Higher than org's 80% } }, "techStack": ["React", "TypeScript"] }

AROG Benefits:

  • Customizes automation per team's tech stack
  • Runs React-specific tests automatically
  • Maintains team autonomy
  • Still follows org-wide policies
